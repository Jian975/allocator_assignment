Goals and Objectives

The goal of this project is to write a complete MIPS program from scratch.

Overview

For this project you will be implementing a memory allocator in MIPS assembly.

Memory Allocator

A memory allocator is a program that gives you memory. For example, in C, you used malloc. Here, you will be implementing your own version of malloc and free in MIPS assembly. The following conditions must be met:
  • Malloc must take in a single number (the number of bytes) and allocate a continguous memory chunk of the closest greater multiple of 4 size.
    • Malloc(4) returns a continguous memory chunk of size 4
    • Malloc(5) returns a continguous memory chunk of size 8
    • Malloc(7) returns a continguous memory chunk of size 8
  • Malloc must return NULL if it doesn't have enough memory to fulfill the allocation request.
  • Free must take in a pointer and freed memory must be available to be allocated again.
  • If you have multiple free chunks of memory touching each other, they must be coalesced into a single large memory chunk.
  • You must have a print function that prints the list of allocated and free memory chunks.

Input Specification

The puzzle will be provided to you on STDIN (i.e., as if a user were typing it in at the console).  The Input will contain multiple lines of ascii text (strings) where each line may have a different meaning.
The first line entered will be a number representing the arena size. The arena size is the number of bytes the allocator has to hand out. The legal range for the arena size is between 4 and 1000, inclusive. If the arena size is not within that range, your program should print the following error message and terminate.

    Error message 1
    Invalid arena size, allocator terminating.

The following lines will be a sequence of commands, with one command per line. {variable} is a character from a-z (note we can have at most 26 variables). {number} represents a digit string. Here's the list of commands (the commands are guaranteed to be in this format):

  • m {variable} {number}
    • m for malloc
    • Allocate a memory chunk of {number} bytes and store it in {variable}
  • f {variable}
    • f for free
    • Free {variable}
  • p
    • p for print
    • Print the memory list
  • g {variable}
    • g for get
    • Print the value of {variable} to STDOUT
  • s {variable} {number}
    • s for set
    • Sets {variable}={number}

Input Processing Hint

You will need to read the numberic input as strings and convert them to integer by writing your own ascii-to-int function.

Memory List

The memory list will be have one element per line, and each element will have the format:
"[size={number}, allocated={1 for yes, 0 for no}, address={16 digit address relative to start of arena}]"
For example, if your arena size is 512 bytes, and you haven't made any allocations, the memory list will look like:
"[size=512, allocated=0, address=0000000000000000]"
The memory list will change as you allocate and free memory. For example, after allocating 4 bytes, your memory list will look like:
"[size=4, allocated=1, address=0000000000000000]"
"[size=508, allocated=0, address=0000000000000004]"
For more information, look at the example input and output files.

Algorithm

For allocating memory, you will need to use the best fit strategy, which means you have to return the memory chunk that best fits the requested size. Suppose you have two free memory chunks: memory chunk 1 has a size of 4 and memory chunk 2 has a size of 100. If the user requests 4 bytes of memory, you must return memory chunk 1.

You don't always have a perfect fit, which means you will need to split free memory chunks. Going back to the example in the last paragraph, suppose memory chunk 1 has a size of 8 instead of 4. Now, if the user requests 4 bytes of memory, you must only return the first 4 bytes of memory chunk 1. To do this, you must add a new free memory chunk entry to your memory list of size 4 for the second half of memory chunk 1 that wasn't used. You may assume the user will only use the first 4 bytes of memory chunk 1 (you don't need to worry about buffer overflow).

When freeing memory, you will need to find the entry in your memory chunk associated with the memory chunk being freed and mark it as free. You must coalesce freed memory, which means if there are other free memory chunks touching the newly freed memory chunk, you must merge them. For example, if you have [free][allocated][free], and you free the second memory chunk, your memory will look like [free] instead of [free][free][free].

Tips and Tricks

You will likely need to store metadata like chunk size somehow. You might be tempted to store metadata right in front of the memory chunk, but don't, since it will mess up the relative memory addresses required by the print memory list command. Instead, consider storing the metadata outside the arena.

Since we have at most 26 variables, we could have at most 43 memory chunks at a time (26 allocated chunks and 27 free chunks between the allocated chunks). This means we have a static upper bound for the amount of space we need for metadata storage, which means you can use an array to store metadata.

Avoid using a linked list since you do not have access to malloc and free (as you are implementing them right now), which means it's nontrivial to add or remove an entry from a linked list. It is likely easiest to limit yourself to arrays in terms of data structures.

The entry point to your program must be a global symbol named main, or else your program won't link correctly into a load module and the simulator will refuse to run it.

To simplify assembling and linking your program, you can use the Makefile from experiment 5 as a template for creating a Makefile for this project by replacing all occurrences of that experiments object (.obj) and executable (.out) files with the ones for your solution.

Make intelligent use of functions - e.g., write functions for reading input, writing output, etc. If you write your output routines early, you can use them for debugging as you write the rest of your code.

Test early and often.

When you have a bug, use the simulator's built-in debugger to step through your code. Run the simulator with the -d option to enable the debugger, and the -t option to turn on line-by-line tracing of instructions. Be sure to read the using rbug section in the RSIM Reference Manual to see what commands are available; in particular, breakpoints, single-stepping, and register commands will be useful.

You can, if you wish, put all of the functions implementing your solution in the allocator.asm file. However, just as in programming using high-level languages, good programming practice would be to separate out the parts of your solution into major functional units, and to put the code for each unit in a separate source file.
